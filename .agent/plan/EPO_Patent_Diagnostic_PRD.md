# 欧洲专利（EPO）创新性智能诊断系统 PRD

> 战略方向：体验与留存驱动型 (Experience & Retention Driven)
> 最后更新：2026-05-20

## 🎯 阶段一：战略定位与成功基准 (Why & Who)

### 1. 产品愿景与战略语境
为国内有出海需求的企业与代理机构提供**极简、高透明度**的欧洲专利出海创新性评估服务。系统通过跨语种比对分析，帮助客户在投入高昂的跨国申请费前，准确预判在欧洲专利局（EPO）的授权概率，降低出海试错成本，增强平台专业信任感。

### 2. 重塑的问题陈述 (Problem Statement)
*   **传统误区**："我们需要给客户做一个专利检索网站。"
*   **重塑后**："中国申请人在向欧洲提交专利时，因存在严重的语言壁垒及对EPO检索规则不熟悉，无法预判其技术在欧洲的'新颖性'与'创造性'。盲目申请极易遭遇由于被驳回而带来的巨大资金与时间沉没成本。客户亟需一种低认知摩擦的方式，提前看到阻碍其授权的海外现有技术。"

### 3. 量化的成功标准 (Success Metrics)
*   **转化与留存**：首月使用服务生成报告的用户，次月复用率（复购/再次评估）需达到 **25%** 以上。
*   **交付价值验证**：PDF诊断报告下载率需达到 **80%** 以上。
*   **认知摩擦降低**：在等待分析的"解析加载页面"跳出率（Drop-off rate）需控制在 **5%** 以内。

### 4. 边界界定 (Out of Scope)
*   不包含美国（USPTO）、日本、PCT等其他国家和地区的专利检索。
*   不包含对欧洲专利申请文本的自动机器翻译生成（不代写）。
*   不包含商标、外观设计专利的分析。

## 🧩 阶段二：核心模块与执行漏斗 (What)

### 1. 功能名称与需求描述
**EPO 出海概率探测引擎 (EPO Probability Engine)**
允许用户上传中文专利文档（PDF/Word），系统自动解析并跨语种提取技术特征，在欧洲专利库中寻找阻碍其授权的核心"对比文件"，计算出申请通过概率，并在前端提供交互式的对比看板以及PDF导出功能。

### 2. 相关页面设计与信息层级
*   **页面 A：极简探测舱 (Upload Hub)** - 中心化拖拽上传区，明确提示"上传中文文本，一键探测欧洲授权概率"。
*   **页面 B：全息思考面板 (Thinking Indicator)** - 动态流式展示底层处理节点："提取中文权利要求 -> 跨语种语义映射 -> 检索 EPO 数据库 -> 冲突特征比对中..."，缓解等待焦虑。
*   **页面 C：诊断交互看板 (Diagnostic Dashboard)** - 主体分屏视图。左侧展示中文权利要求，右侧展示欧洲"对比文件"。通过高亮色块进行"一对一连线"映射。
*   **页面 D：一键导出中心 (Export Action)** - 悬浮按钮，生成结构化、带企业水印的正式 PDF 诊断报告。

### 3. 用户旅程 (User Journey)
1. **上传触发**：拖入中文专利交底书/PDF。
2. **视觉安抚**：进入思考面板，看着系统依次亮起解析、检索、比对的节点灯。
3. **结果触达**：约 1-3 分钟后，切入看板，看到通过概率评分。
4. **深度探索**：点击对比文件，双屏视图自动高亮显示冲突特征，瞬间理解新颖性不足原因。
5. **离线决策**：下载精美PDF报告，用于内部分享与决策。

## ⚖️ 阶段三：验收标准 (Acceptance Criteria)

### 1. 核心流程验收 (BDD 语法)

**场景一：正向探测流程**
*   `GIVEN` 用户登录系统，停留在“极简探测舱”页面
*   `WHEN` 用户拖拽上传一份格式为 .pdf、大小为 5MB 的中文专利文档
*   `THEN` 系统必须拦截界面跳转，展示“全息思考面板”动效
*   `AND` 评估完成后，必须自动重定向至“诊断交互看板”，显示概率评分和对比文件列表

**场景二：负向防御机制（文件不合规）**
*   `GIVEN` 用户停留在“极简探测舱”页面
*   `WHEN` 用户试图上传 .jpg 图片或超过 50MB 的文档
*   `THEN` 前端系统必须在 0.5 秒内拒绝上传并拦截
*   `AND` 抛出红色Toast提示：“格式不受支持或体积过大”，绝不允许进入加载页面

**场景三：极端边界场景（无对比文件）**
*   `GIVEN` 用户的专利属于极端前沿的“无人区”技术
*   `WHEN` 系统完成 EPO 数据库检索，命中对比文件数量为 0
*   `THEN` 看板的通过概率必须显示为 “99%（极高）”
*   `AND` 诊断理由提示“未检索到高相似度技术，建议尽快提交”，并隐藏对比分屏视图

### 2. UI 与交互强制检查清单
- [ ] **高亮映射规则**：在双屏视图中，点击右侧英文段落时，左侧对应中文段落必须在 0.2 秒内滚动至居中并高亮。
- [ ] **PDF 导出规格**：必须包含企业防伪水印，A4 纵向排版，必须包含目录页与评估结论摘要页。
- [ ] **状态保持**：在“思考面板”加载中途刷新网页，系统必须恢复当前分析进度，绝不允许要求用户重新操作或重新扣费。

## ⚙️ 阶段四：隐形骨架与数据内核 (NFR & Data)

### 1. 非功能性需求 (NFR)
*   **🔐 极高机密性约束 (Security)**：交底书在推理提取时必须脱敏。物理文件在服务器留存时间不得超过 24 小时（或生成报告后立即物理销毁）。
*   **⚡ 性能极限 (Performance)**：单份文档（10万字内）端到端评估时间**不得超过 3 分钟**。若超时，必须在前端降级为异步模式（“完成后短信/邮件通知”）。

### 2. 数据追踪矩阵 (Tracking Plan)
| 事件名称 (Event Name) | 触发条件 (Triggers) | 属性参数 (Properties) | 商业映射 (Business Mapping) |
| :--- | :--- | :--- | :--- |
| `analysis_started` | 成功接收文件并调用解析 API | `file_size`, `tech_domain` | 追踪分析漏斗起点 |
| `analysis_completed` | 概率评分和看板成功渲染 | `processing_time`, `final_score` | 监控系统性能及模型响应成功率 |
| `comparison_interacted` | 点击侧边栏具体对比文件时 | `prior_art_id`, `click_depth` | **核心留存指标**：衡量双屏视图真实使用率 |
| `report_downloaded` | 成功生成并下载 PDF 报告 | `page_count` | 衡量服务的最终业务闭环和交付价值 |
